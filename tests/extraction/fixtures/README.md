# Extraction test fixtures

This directory holds **synthetic** firmware binaries used by the
extraction test suite. Every binary here is generated by a Python
builder living in the same directory, and the builder is committed
alongside the binary so the binary can be regenerated.

## What's here

| Builder | Output | Purpose |
| - | - | - |
| `synthetic_uefi_volume.py` | `~16 KiB` PI volume + 1 FFS file | UEFI PI volume / raw FFS extractor tests |
| `synthetic_option_rom.py` | `~2 KiB` two-image PCI option ROM | Option ROM extractor tests |
| `synthetic_microcode.py` | `~4 KiB` two-blob Intel microcode | Microcode extractor tests |

The `golden/` subdirectory is reserved for snapshot binaries used by
`tests/extraction/test_golden.py` (added in task 23). Until then it
stays empty.

## How to use real-world binaries locally

Real firmware images aren't checked in — vendor confidentiality, file
size, and license terms make that a non-starter. To test against
real-world binaries on your own machine:

1. Drop the binary into a path **outside** the repo
   (e.g. `~/firmware-corpus/intel-x1g11.rom`).
2. Set the env var `LOKI_EXTRACTION_REAL_FIXTURE=/abs/path/to/binary`
   before running the extraction tests.
3. Tests that opt into real-world fixtures consult
   `request.config.getoption("--real-fixture")` or the env var; they
   skip when neither is set so CI runs stay green.

Future work tracked under `.kiro/specs/extraction-pipeline/design.md`
"What's deliberately not tested" lists public corpora (linux-firmware,
firmware analysis CTF binaries, etc.) suitable for ad-hoc validation.

## How to regenerate a fixture

When you intentionally change a fixture's shape (e.g. add a new
section to the synthetic UEFI volume), the corresponding `golden/`
snapshot becomes stale. Bump the version in the filename
(`uefi_volume_v1.bin` → `uefi_volume_v2.bin`) and check both files in.
Don't overwrite the old fixture — keep the history so older spec
revisions can be rebuilt.
