# Baseline persistence test fixtures

This directory holds **synthetic** baseline records used by the
baseline-persistence test suite. Every record is generated by a
Python builder living in the same directory, and the builder is
committed alongside the snapshot so the snapshot can be regenerated
deterministically.

## What's here

| Builder | Output | Purpose |
| - | - | - |
| `synthetic_baseline.py` | `BaselineRecord` with N classifications | Deterministic input for round-trip + golden tests |

The `golden/` subdirectory holds checked-in snapshots used by
`tests/baseline/test_golden.py`. The current entries are:

- `canonical_v1.yaml` — full Baseline_File as written by
  `BaselineStore.save()`. Compared modulo the
  `written_by_extractor_version` envelope field, which is normalized
  to `"loki-GOLDEN"` so version bumps don't invalidate the snapshot.
- `canonical_v1.json` — the re-loaded baseline payload after
  stripping envelope fields. Compared as a Python dict for clean
  diff output on regression.

## Determinism contract

The synthetic builder uses `uuid.uuid5` derivations seeded from the
`(vendor, model, firmware_version)` triple, so two calls with the
same arguments produce byte-identical records. Combined with
`yaml.safe_dump(sort_keys=True)` in the envelope serializer, the
on-disk YAML is deterministic across runs and across machines.

The two volatile fields the snapshot tests have to handle are:

- `written_at` in the envelope — frozen via a `_FrozenDatetime`
  subclass patched into `loki.baseline.store.datetime` for the
  duration of the test.
- `written_by_extractor_version` in the envelope — passed
  explicitly as `"loki-GOLDEN"` via the `written_by` kwarg, so
  the snapshot doesn't drift when the project version is bumped.

## How to regenerate the snapshots

When you intentionally change the synthetic builder's shape (e.g.
add a new field to `BaselineRecord`, change the default
classification count, swap a label) the existing snapshots become
stale.

1. Run the regenerate flag:

   ```bash
   LOKI_REGENERATE_GOLDEN=1 .venv/bin/pytest tests/baseline/test_golden.py
   ```

   The two snapshot tests will skip with a "regenerated" message;
   the round-trip and filename-stability tests still run.

2. Inspect the diffs in `canonical_v1.yaml` and `canonical_v1.json`
   to confirm the change is intentional.

3. Re-run without the env var:

   ```bash
   .venv/bin/pytest tests/baseline/test_golden.py
   ```

   All four tests should pass.

4. If the change is **breaking** for downstream consumers of the
   YAML format (e.g. you renamed an envelope key, bumped
   `SCHEMA_VERSION`, or restructured the baseline payload),
   bump the snapshot filename: `canonical_v1.*` → `canonical_v2.*`
   and check both versions in. Don't overwrite the old snapshot —
   keep the history so older spec revisions can be rebuilt for
   debugging.

5. Commit the updated builder *and* the regenerated snapshots
   together. Reviewers should see the builder change in the same
   commit as the snapshot diff.

## Notes for reviewers

A diff in `canonical_v1.yaml` or `canonical_v1.json` that wasn't
accompanied by a builder change is a regression in either:

- The model layer's `BaselineRecord.model_dump(mode="json")` shape.
- The envelope serializer (`loki/baseline/envelope.py`).
- The `slug()` / `filename_for()` helpers in
  `loki/baseline/naming.py`.
- The PyYAML version's interaction with `safe_dump(sort_keys=True,
  default_flow_style=False)`.

Either case warrants a careful review — these are the contracts the
golden file pins so refactors don't silently corrupt the persisted
data format.
